**★ CORRECTION 2026-07-13 (on-Deck, live app over CDP).** Chapter-aware seek IS buildable on
TVHTML5 — the spike above probed the wrong endpoint. `POST /youtubei/v1/next` (page's TVHTML5
context + `INNERTUBE_API_KEY`, videoId passed directly, `status 200`) returns
`engagementPanels[…panelIdentifier='engagement-panel-macro-markers-description-chapters']` AND a
`macroMarkersListEntity` in `frameworkUpdates.entityBatchUpdate.mutations[].payload`. Its
`markersList` (`markerType: MARKER_TYPE_CHAPTERS`) is an array of markers, each with
`title.simpleText`, **`startMillis`** (string ms), `durationMillis`, and
`onTap.innertubeCommand.seekToVideoTimestampCommand.offsetFromVideoStartMilliseconds`. Verified for
`X5xlTpuiRCE` (chapters "Flying over Italy" @0, "The Amalfi Coast" @348000, "Capri" @533000, …). So
`/next` is the chapter source `/player` omits; combine with the already-verified `#movie_player.seekTo`
to build prev/next-chapter L2/R2 (fixed-interval skip stays the fallback when a video has no chapters).
Details + implementation notes now in durable/input-ux §18.

## Media & VA-API (S0.5) — ★ HW DECODE FIXED ON M138
The m114 blocker is GONE. On m114, `--enable-features=VaapiVideoDecodeLinuxGL --use-angle=gl` engaged
`VaapiVideoDecoder` but ANGLE painted green-band corruption on every frame (broken DMA-buf import of
the tiled NV12 VA surface on radeonsi). On M138 the SAME two flags were re-tested on-Deck (OLED) via
the shipped flatpak (Game Mode, real Leanback VP9 playback), injected with
`flatpak override --user --env=DECKBACK_EXTRA_ARGS='--enable-features=VaapiVideoDecodeLinuxGL --use-angle=gl'`
(launcher appends DECKBACK_EXTRA_ARGS to engine argv, main.cpp:263). Result under the mandatory DUAL
gate:
- decoder = `VaapiVideoDecoder`, `kIsPlatformVideoDecoder=True` (via deckctl + a single-connection capture)
- pixels = **0.0% green-band rows** across 5 sampled frames, visually confirmed clean
  (`tests/deck/lib/probes.py:video_corruption_verdict`, PIL, L0-tested).

So `config/app.json` now **ships HW decode** — the two flags are baked into `cobalt_flags`. Both flags
remain required: `--use-angle=gl` alone gets VA-API but the feature flag selects the decoder; the
feature flag alone leaves ANGLE on Vulkan/radv and silently falls back to software.
**P4 power (2026-07-10, OLED, on battery):** `just power 120` measured **mean 5.91 W over 120 samples
under `VaapiVideoDecoder` — PASS vs ≤9 W** (playback confirmed before AND after; P = V×I, no power_now
node). Both halves of the P4 gate now green. LCD unit unknowable. See `../durable/hardware.md` ★ note.

Gotcha for any decoder probe: a CDP client MUST call `cdp.enable_media()` or the Media domain never
streams `kVideoDecoderName` and the decoder reads as None (a broken probe, not software decode).
