Cobalt **≤25 is classic Starboard Cobalt** (own H5VCC-derived renderer: `cobalt/layout`, `cssom`,
`dom`, `h5vcc`; **no Blink**; root `.gn` → `//starboard/build/config/BUILDCONFIG.gn`). Verified

## Checkout layout (S0.1) — verified
Chrobalt is a Chromium fork: its `DEPS` is keyed under `src/`, so the gclient **solution must be
named `src`** and the engine tree lives at `./cobalt/src` (host) / `/src/src` (container). Run
gclient **from the gclient root** (`/src`) with `gclient config --name src --unmanaged
https://github.com/youtube/cobalt.git` so the managed solution dir isn't itself a bind-mount point —
otherwise gclient does its "conflicting directory → `_bad_scm`" move and silently skips hooks. The
scripts encode this via `COBALT_TREE` / `CTR_TREE` (lib.sh). `gclient sync --no-history` fetches gn +
hermetic clang (llvmorg-17) + Debian bullseye sysroot + Rust. Full checkout ≈ 22 GB.

## Media stack (S0.5) — RESOLVED 2026-07-08
**Decode path = Chromium's media pipeline. Starboard `SbPlayer` is not used at all.** The built
`content_shell` contains `VaapiVideoDecoder`, `VaapiWrapper`, `FFmpegDemuxer`, `Dav1dVideoDecoder`
and **zero** `SbPlayerCreate`/`starboard` symbols; the live player reports `kRendererName=RendererImpl`.
Phase 4 therefore targets Chromium's VA-API stack, not `SbPlayer`.

### VA-API hardware decode WORKS — with exactly two flags
    --enable-features=VaapiVideoDecodeLinuxGL --use-angle=gl

Measured on the Deck against real Leanback playback (CDP `Media` domain, the same data
chrome://media-internals shows):

| flags | kVideoDecoderName | kIsPlatformVideoDecoder | served |
|---|---|---|---|
| none (what `just run` ships today) | `Dav1dVideoDecoder` | **false** | AV1 854×480 |
| `VaapiVideoDecodeLinuxGL` only | `Dav1dVideoDecoder` | **false** | AV1 854×480 |
| `VaapiVideoDecodeLinuxGL` + `--use-angle=gl` | **`VaapiVideoDecoder`** | **true** | VP9 1280×720, 0/84 dropped |

- **`--use-angle=gl` is required.** The default backend is ANGLE-on-Vulkan (`radv`); VA-API does not
  engage there and silently falls back to software. With `--use-angle=gl` the GL renderer becomes
  native `radeonsi` Mesa GL and `VaapiVideoDecoder` is selected. Cost: we give up the Vulkan backend —
  **needs a render-perf/battery check under gamescope before adopting** (Phase 4 gate).
- **`--ignore-gpu-blocklist` and `VaapiIgnoreDriverChecks` are NOT needed.** Adding them changes
  nothing about decoder selection; avoid them (they suppress real driver checks).
- **Today we ship software decode.** Baseline playback selected `Dav1dVideoDecoder` — i.e. YouTube is
  serving **AV1** and we decode it on the CPU. That is the worst case for the P4 battery gate.

#### UPDATE 2026-07-10 — the two flags are now in the shipped config, and VA-API is verified end to end
The finding above was a spike result; the flags had **never been in `config/app.json`**, so every
on-Deck run until today still selected software decode. Corrected: `cobalt_flags` now carries
`--enable-features=VaapiVideoDecodeLinuxGL` and `--use-angle=gl`. Verified through the shipped path
(launcher → content_shell → real Leanback playback of a guest video), OLED, 2026-07-10:

    deckctl decoder --require-vaapi  ->  VaapiVideoDecoder, kIsPlatformVideoDecoder=True   (exit 0)
    1280x720 VP9, 2 dropped / 1625 total (0.12%), 0 corrupted

So AV1 steering (VP9 served, not AV1) **and** hardware decode both hold in the config we deploy — not
just in a hand-flagged spike. The `just power` battery number that this unlocks is still unmeasured
(the P4 gate now refuses to sample on AC or with the panel off; see durable/harness.md F16).

#### UPDATE 2026-07-10 (later) — VA-API decode is VISUALLY CORRUPT; the two flags are REVERTED
The "verified end to end" above was a **false pass**, and the way it passed is the important part.
A user watching the shipped app reported *"on every video I can see transparent green lines — buggy
artifacts."* A CDP screenshot during VP9 720p playback confirmed it: heavy **green horizontal
banding** with orange/tan ghosting over the decoded frame (scratchpad frames captured this session).
`kVideoDecoderName=VaapiVideoDecoder`, `kIsPlatformVideoDecoder=true`, `corruptedVideoFrames=0`,
`droppedVideoFrames=0` — **every automated signal we had said PASS while the panel was striped.**
`corruptedVideoFrames` counts decoder-reported corruption; this corruption is in ANGLE's *import* of
a correctly-decoded surface, so nothing in the media pipeline flags it. This is a textbook F1
("a check that cannot fail is not a check") miss: decoder-name + corrupted-frame checks cannot see
display-layer corruption. Only a **pixel/screenshot check** can.

Root cause: ANGLE's DMA-buf / EGLImage import of the **tiled NV12 VA surface** is broken on this
Mesa/RADV + Van Gogh/Sephiroth stack — the alternating green bands are the classic wrong-tiling /
wrong-stride import (tiled surface read as linear). It is a GL-backend interop bug, not a decode bug.

7-variant on-Deck flag bisection (OLED, all VP9 720p, `aqz-KE-bpKQ`), `decoder` via CDP Media domain,
image via CDP screenshot:

| extra flags (on top of / overriding config) | decoder | image |
|---|---|---|
| *(shipped: `VaapiVideoDecodeLinuxGL` + `--use-angle=gl`)* | VaapiVideoDecoder (HW) | **green bands** |
| `--disable-features=UseMultiPlaneFormatForHardwareVideo` | VaapiVideoDecoder (HW) | green bands |
| `--disable-gpu-memory-buffer-video-frames` | VaapiVideoDecoder (HW) | green bands |
| `--use-angle=gles` | VaapiVideoDecoder (HW) | green bands |
| `--enable-features=…,UseMultipleOverlays --enable-hardware-overlays=…` | VaapiVideoDecoder (HW) | green bands (no overlay promotion under X11 ozone) |
| `--use-angle=vulkan` | VpxVideoDecoder (SW) | **clean** |
| `--disable-gpu-compositing` | VpxVideoDecoder (SW) | **clean** |

The corruption is **inseparable from `VaapiVideoDecoder`**: it survives every frame-buffer and
overlay lever, and every configuration that renders clean has silently reverted to **software**
`VpxVideoDecoder`. HW decode on this stack ⇒ corrupt; clean ⇒ software. There is no flag-level config
that is both hardware and clean.

**Action taken:** removed `--enable-features=VaapiVideoDecodeLinuxGL` and `--use-angle=gl` from
`config/app.json`. We ship **software decode** (VpxVideoDecoder, clean image) — the state the app had
before these flags landed, which is why no user reported green lines until this session added them.
P4 hardware decode is **blocked**, not done. Re-enabling requires fixing the ANGLE↔VA-API import at
the engine/Mesa layer (or a working hardware-overlay path under gamescope), and any re-enable MUST be
gated on a **pixel check**, never decoder-name/corrupted-frame alone.

Bisection method (reproducible): `DECKBACK_EXTRA_ARGS` is appended after `cobalt_flags`, and Chromium
switch parsing is last-wins for valued switches, so a variant can override `--use-angle` / merge into
`--enable-features` from the launch env without a rebuild — the scratchpad `relaunch.sh` +
`variant.sh` drive it (relaunch needs `DISPLAY=:0 XDG_RUNTIME_DIR=/run/user/1000` from the gamescope
session, else content_shell dies "Missing X server or $DISPLAY").

### Do not trust `SystemInfo.getInfo` → `gpu.videoDecoding`
It stays **empty even when VA-API is demonstrably decoding**. That field is the *legacy* VDA profile
list; the modern mojo `VideoDecoder` path (which VA-API uses) reports through
`supported_video_decoder_configs` instead. The reliable signal is the CDP `Media` domain's
`kIsPlatformVideoDecoder` / `kVideoDecoderName` on a live player. An early read of `gpu.videoDecoding`
nearly produced a false "no hardware decode on this hardware" conclusion.

`VAEntrypointVLD` (decode) profiles exposed:

    MPEG2Simple, MPEG2Main, VC1Simple/Main/Advanced, H264ConstrainedBaseline/Main/High,
    HEVCMain, HEVCMain10, JPEGBaseline, VP9Profile0, VP9Profile2, **AV1Profile0**

## Input mechanism (S0.6) — RESOLVED 2026-07-08
**Leanback is fully keyboard/remote navigable.** Verified on the Deck via CDP `Input.dispatchKeyEvent`
against the onboarding UI: `ArrowDown` moved focus "Get started" → "Terms & Privacy", `ArrowUp` back,
`Enter` activated the button and advanced to the next screen. CDP-dispatched key events are **trusted**
(`isTrusted=true`), so Leanback's real handlers fire (JS-synthesized events would not). `scripts/cdp.py`
gained a `--keys` mode (KEYMAP + `Input.dispatchKeyEvent`, focused-element probe) used for this.

**Phase-3 mechanism = translate gamepad → key events.** Two implementation options (decision pending):
- **(A, plan default)** a `patches/` quilt change: evdev→`ui::KeyEvent` in Cobalt's Ozone/browser input.
  Native, no debug-port dependency; costs a patch to rebase each Cobalt bump.
- **(B, revealed viable)** the **launcher** reads the gamepad evdev (libevdev) and dispatches keys over
  the existing CDP bridge (`Input.dispatchKeyEvent`) — NO Cobalt patch, keymap stays in `config/app.json`
  (remotely overridable, a plan design goal), reuses the DevTools client. Depends on the remote-debugging
  port (already on for Phase 6). Loopback CDP latency is sub-ms. The S0.6 spike proves (B) works today.

Under Steam Input (Game Mode) the app sees a **virtual `Microsoft X-Box 360 pad` (`/dev/input/event10`)**,
not the raw `Valve Software Steam Deck Controller` (event4/event9) — so the evdev reader targets the
Xbox-360 pad; the physical→virtual mapping is Steam Input's job (`config/steam_input.vdf`).

## Input on-Deck verification (P3) — CONFIRMED 2026-07-08
The full chain **evdev read → key translation → CDP `Input.dispatchKeyEvent` → Leanback handler** runs
on the Deck. Verified two independent ways against the `deck`/qa build deployed to `~/cobalt-yt/`:

1. **Physical controller.** Real presses produced `input: ArrowUp` / `input: ArrowLeft` /
   `input: A -> Enter` / `input: B -> Escape` in `deckback.log`.
2. **Synthetic pad.** A uinput device advertising `BTN_SOUTH` (so `is_gamepad()` accepts it) injected
   D-pad Down ×2 then `BTN_SOUTH`. Launcher logged `ArrowDown`, `ArrowDown`, `A -> Enter`; CDP showed
   `document.activeElement` move off "Get started" and then **open the Terms & Privacy overlay**.
   `/dev/uinput` is writable by `deck` (ACL), so this test needs no root.

Device facts (this Deck, Steam Input active):
- **Only `event10` (`Microsoft X-Box 360 pad 0`) advertises `BTN_SOUTH`.** The raw
  `Valve Software Steam Deck Controller` nodes (`event4`, `event9`) do **not** — confirmed by reading
  the `B: KEY=` bitmap in `/proc/bus/input/devices` (bit 0x130). So `is_gamepad()` skipping them is
  correct behaviour, not a bug.
- Attaching a second gamepad makes Steam Input auto-create a **wrapper** pad (`event18`,
  `Microsoft X-Box 360 pad 1`). The launcher opened all three and merged them without duplicate keys.
- When a device disappears, the `lost` branch of `GamepadInput::loop()` rescans and recovers.

### Host bind-mount ownership trap (bit us twice)
`docker-compose.yml` bind-mounts `${SCCACHE_HOST_DIR:-./.sccache}` → `/cache/sccache` and
`${FLATPAK_HOST_DIR:-./.flatpak-cache}` → `/cache/flatpak`. Docker creates a **missing** host dir as
`root:root`; the container's `dev` user is uid 1000 and cannot write it. Failure modes differ:
sccache **silently misses every compile** (no error), while `just flatpak` dies loudly with
`mkdir(/cache/flatpak/share): Permission denied`. `bootstrap.sh` now pre-creates both as the host user.

## Launcher UA injection + navigation (CDP) — RESOLVED 2026-07-08
`just run` (`deckback-launcher`) now drives content_shell to interactive Leanback unaided. Chosen
mechanism: **startup CDP override** (the "b" option from S0.2), implemented in `launcher/src/navigator.*`:

  the TCP socket open despite a `Connection: close` request header. A read-until-close client blocks
  to its poll timeout and fails *every* discovery — this silently kept BOTH the navigator and the
  Phase-6 player from ever attaching to a real engine (only the fake test server, which closes, hid
  it). `launcher/src/devtools.cpp:http_get` now returns as soon as the Content-Length body arrives.
- **content_shell (M114) supports multiple concurrent CDP clients per page target.** The navigator
  and player each hold their own `DevToolsClient` to `:9222` simultaneously without contention.
- Dev-only engine flags (`--no-sandbox`, pre-zypak) are passed via `DECKBACK_EXTRA_ARGS` (set by
  `run.sh`) so they never land in the shipped `config/app.json`.

## youtube.com/tv enforces Trusted Types — CDP innerHTML injection was a silent no-op — RESOLVED 2026-07-10
The Leanback document ships `require-trusted-types-for 'script'`. Any `element.innerHTML = "<...>"`
throws `This document requires 'TrustedHTML' assignment` and renders **nothing** — no error surfaces
to the launcher, because a failed overlay must never break the action it announces, so the failure is
by-design invisible. Consequences, all confirmed on-Deck:

- The controls card (`onboarding.cpp`, `innerHTML`) rendered in every host-side test and **nothing**
  on the real page. It had never been seen on hardware because the account gate (below) masked
  Leanback until now.
- The touch-lock toast (`overlay.cpp`) was **unaffected** — it uses `textContent`, which Trusted
  Types does not gate. That is why its test passed while the card's failed.

**Fix:** `js_trusted_html()` in `overlay.cpp` wraps HTML in a memoised Trusted Types policy
(`trustedTypes.createPolicy('deckback', {createHTML: s => s})`), falling back to the raw string where
Trusted Types is absent (about:blank, so the error page is unchanged there). youtube.com/tv does
**not** pin a `trusted-types` allowlist, so an arbitrary policy name is accepted (probed: `deckback`,
`dompurify`, `default`, `goog#html` all create). Verified on-Deck: policy-wrapped `innerHTML` yields
`card present: true` on the live Leanback page. `textContent` and `setAttribute` never needed this;
only `innerHTML` / `outerHTML` / `insertAdjacentHTML` do.

The L2 test that checks this (`test_ux.py::test_controls_card_can_render_over_leanback`) re-creates
the launcher's injection inline and had drifted to a bare `innerHTML` — so it failed on the real page
while the fixed launcher worked. It now shares `probes.trusted_html_js`, which MUST mirror
`js_trusted_html`. A test that hand-copies the product's technique is only as good as the copy.

**Bump note:** re-confirm the CSP header on youtube.com/tv on any bump; if Google adds a
`trusted-types` allowlist, the `deckback` policy name may start throwing and the fallback (raw string)
would silently break the card again.

## The account gate masks Leanback, and the `leanback` fixture cannot tell them apart — 2026-07-10
On a fresh/guest profile every route lands on an account chooser — **"Add account | Add a kid account
| Watch as guest"** — and stays there until dismissed. `location.href` is `youtube.com/tv`, a
`<video>` element exists (idle, `readyState 0`), and arrow keys move focus between the three tiles, so
the gate passes every shallow "are we on Leanback?" check. The `leanback` pytest fixture only asserts
the URL, so it happily ran the entire L2 suite **against the account gate**, not the app:
`test_hardware_decode_is_vaapi` saw no decoder (nothing playing), `test_uinput_button_a_activates` saw
A open a sign-in pane instead of a video. This is the same "Watch as guest swallows every route" trap
noted for the deep-link test (TASKS P2, 2026-07-08), now shown to poison the whole suite.

To reach real playback: focus the tile whose text matches `/watch as guest/i` and press a **trusted**
Enter (`ArrowRight` alone will not cross all three tiles reliably; focus the element directly). Then a
`#/watch?v=<id>` deep link plays. The fixture should gain a gate-dismiss step, or a deeper liveness
probe than "URL is youtube.com/tv", before the suite can be trusted unattended — filed against
`tests/deck/conftest.py`.
